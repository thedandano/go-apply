# Implementation Plan: T1 Category-Aware Skills Edits

**Branch**: `005-fix-t1-categorized-skills` | **Date**: 2026-04-25 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/005-fix-t1-categorized-skills/spec.md`

## Summary

Add an optional `category` field to `port.Edit` so that T1 skill edits can target a specific category when the resume's skills section is `kind="categorized"`. Update the apply-edits service to route category-targeted ops (add/replace) with comma-split value parsing, reject missing/unknown categories with actionable messages, and preserve the existing flat path unchanged. Update the MCP tool description string and workflow prompt so the orchestrator constructs valid category-targeted edits.

Technical approach: minimal surgical change across three layers (port → service → adapter). No schema version bump, no sidecar format change, no new interfaces.

### Agent A — Implementation Notes

#### port/tailor.go

Add one field to `Edit`:
```go
Category string `json:"category,omitempty"`
```
No interface changes. `port.Tailor.ApplyEdits` signature unchanged.

#### apply_edits.go — `applySkillsEdit`

Replace the categorized guard block:

```
// BEFORE: hard-reject all ops when categorized
if sections.Skills.Kind == model.SkillsKindCategorized {
    return fmt.Sprintf("op %q on skills requires kind=%q ...", ...)
}

// AFTER: route by category
if sections.Skills.Kind == model.SkillsKindCategorized {
    cats := sections.Skills.Categorized
    if edit.Category == "" {
        return fmt.Sprintf("op %q on categorized skills requires a category; available: [%s]",
            edit.Op, sortedKeys(cats))
    }
    if _, ok := cats[edit.Category]; !ok {
        return fmt.Sprintf("category %q not found; available: [%s]",
            edit.Category, sortedKeys(cats))
    }
    switch edit.Op {
    case port.EditOpAdd:
        cats[edit.Category] = append(cats[edit.Category], splitTrim(edit.Value)...)
    case port.EditOpReplace:
        cats[edit.Category] = splitTrim(edit.Value)
    default:
        return fmt.Sprintf("unsupported op %q for section %q", edit.Op, edit.Section)
    }
    return ""
}
```

Two helpers needed:
- `sortedKeys(m map[string][]string) string` — returns sorted, comma-joined keys
- `splitTrim(s string) []string` — splits on `,`, trims whitespace, drops empty tokens

#### apply_edits_test.go

The existing test `"skills categorized kind rejects flat ops"` currently asserts that a no-category edit is rejected. After the fix, this test remains valid BUT the assertion must change:
- **Rename to**: `"skills categorized rejects ops with missing category"`
- **Update assertion**: check that the rejection reason contains both `"requires a category"` AND `"available:"` (not the old `"categorized not supported"` text)
- Keep the existing structural assertions: `Categorized["Cloud"]` still has len==2, `Flat == ""`

Add a second new test immediately after:
- **Name**: `"skills categorized rejects ops with unknown category"`
- **Input**: edit with `Category: "Nonexistent"` on a categorized skills section
- **Assertions**: rejection reason contains `"category \"Nonexistent\" not found"` AND `"available:"`, and no category's contents change

New test cases to add:
1. `add` on categorized with valid category → items appended individually
2. `replace` on categorized with valid category → category list replaced
3. `add` on categorized with missing category → rejected with available categories in message (covered by renamed existing test)
4. `add` on categorized with unknown category → rejected with available categories in message (covered by new test above)
5. `category` field on flat section → silently ignored, edit applied normally
6. `add` on categorized with comma-separated value → split into individual items (not one entry)
7. Empty categorized map → rejection message notes no categories exist

### Agent B — Implementation Notes

#### server.go

Update the `mcp.WithDescription(...)` for `submit_tailor_t1`'s `edits` parameter:

**Current**:
```go
mcp.WithString("edits", mcp.Description(`JSON array of {"section":"skills","op":"replace|add","value":"..."} objects. Max 5 entries. e.g. [{"section":"skills","op":"add","value":"GCP"}]`), mcp.Required()),
```

**New**:
```go
mcp.WithString("edits", mcp.Description(`JSON array of {"section":"skills","op":"replace|add","value":"...","category":"..."} objects. Max 5 entries. "category" is required when skills.kind="categorized" — use a key from sections.skills.categorized. e.g. flat: [{"section":"skills","op":"add","value":"GCP"}] categorized: [{"section":"skills","category":"Cloud","op":"add","value":"GCP, Azure"}]`), mcp.Required()),
```

#### prompt.go

Two changes in `workflowPromptText`:

1. **Tool table row** — update the `submit_tailor_t1` entry to show `category?` in the edit schema:
   ```
   | submit_tailor_t1 | Apply structured edits to the Skills section and rescore | session_id (req), edits (JSON array of {section:"skills", op, value, category?}, req) |
   ```

2. **Step 5 T1 instruction** — expand the call-to-action to distinguish flat vs categorized:
   ```
   2. Call submit_tailor_t1 with edits (max 5 items). Section must be "skills".
      - If skills_section.kind == "flat": omit category.
        e.g. {"section":"skills","op":"replace","value":"AWS, GCP"}
      - If skills_section.kind == "categorized": include category matching a key from
        sections.skills.categorized (returned by submit_keywords).
        e.g. {"section":"skills","category":"Cloud","op":"add","value":"AWS, GCP"}
      Prefer one-for-one replace over pure add to keep section length stable.
   ```
